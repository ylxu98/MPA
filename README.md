## 2. Deployment, Testing, and Experimental Data Export Commands

### 2.1 Environment Setup

```bash
cd mpa-evm-audit

# Install dependencies (first time)
npm install

# Compile contracts
npx hardhat compile
```

### 2.2 Deploy Contracts

```bash
# One-click deployment: Inject BLS shim → Deploy 4 contracts → Generate SRS off-chain → On-chain initialization
npx hardhat run scripts/deploy_all.js --network hardhat

# Inject BLS precompile shim separately (usually called automatically by deploy_all)
npx hardhat run scripts/setup_bls_precompiles.js --network hardhat
```

### 2.3 Run Experiments

```bash
# Run all three major experiments (Auto deploy + test + export CSV)
npx hardhat test

# Run specific experiments individually
npx hardhat test --grep "RANDAO"        # Experiment 1: RANDAO overhead (200 executions)
npx hardhat test --grep "StorageGas"    # Experiment 2: Storage Gas overhead
npx hardhat test --grep "BatchAudit"    # Experiment 3: Large-scale batch auditing T-sweep
npx hardhat test --grep "篡改"          # Security: Tamper detection
```

### 2.4 Export Experimental Data

Once the experiments are completed, CSV files are automatically exported to the results/ directory:

```bash
# View experimental results
cat results/randao_overhead.csv     # RANDAO overhead data
cat results/storage_gas.csv         # Storage Gas overhead data
cat results/batch_audit.csv         # Batch auditing performance data
cat results/gas-report.txt          # Method-level Gas report
cat results/deployed_addresses.json # Contract addresses
```

## 3. Logical Mapping Between EVM Engineering and Python SS512 Simulation Code

### 3.1 Cryptographic Curve Adaptation

| Dimension | Python Simulation (main.py) | EVM On-Chain Engineering |
|------|----------------------|-------------|
| Pairing Curve | SS512 (BN254, Type-I Symmetric) | BLS12-381 (Type-III Asymmetric) |
| Pairing Type | G × G → G_T (Symmetric) | G1 × G2 → G_T (Asymmetric) |
| SRS Structure | pk = (g, g^α, ..., g^{α^{s-1}}) all in G | G1 segment: [g1]^{α^i}; G2 segment: [g2]^α |
| Tag σ | σ = g^{f(α)} ∈ G | σ = [g1]^{f(α)} ∈ G1 |
| Witness w | w = g^{h(α)} ∈ G | w = [g1]^{h(α)} ∈ G1 |
| Pairing Equation | e(σ·g^{-val}, g) = e(w, g^α·g^{-r}) | e(A, g2) = e(w, C), A=σ·g1^{-val}∈G1, C=[g2]^α·g2^{-r}∈G2 |
| Precompile | Calculated within the charm-crypto library | EIP-2537 (0x0f PAIRING_CHECK) |

### 3.2 Module Mapping

| Python Module (main.py) | EVM Engineering | Description |
|----------------------|---------|------|
| \`Poly\` Class | \`test/lib/bls.js\` Poly Class | Polynomial on Fr: evaluate / add / mulScalar / divXMinusR |
| \`DynamicMerkleTree\` | \`bls.js\` buildMerkleTree + contract \`_verifyMerklePath\` | Off-chain tree building, on-chain validation |
| \`Blockchain\` Class | 4 Solidity Contracts | On-chain verification logic split into 4 contracts |
| \`Blockchain.verify_batch()\` | \`BatchAudit.verifyBatch()\` | Single global pairing + Merkle path validation |
| \`User\` Class storage | \`bls.js\` storagePhase + \`FileRootStorage.uploadRoot\` | Off-chain Encryption/Tag, On-chain Root storage |
| \`User\` Class challenge_gen | \`RandAO.generateChallenge\` | RANDAO two-phase replaces Python random numbers |
| \`CSP\` Class response | \`bls.js\` responsePhase | Off-chain CSP response (unchanged) |
| \`aggregate_all_proofs()\` | \`bls.js\` aggregateGlobal | Global aggregation (off-chain verifier) |
| Parameters T/N/n/s/c | \`config.json\` experiment section | Fully aligned |

### 3.3 Core Verification Equation Comparison

**Python (main.py)**:
\`\`\`python
# Type-I Symmetric Pairing
e(global_agg_tag * g^(-val), g) == e(w, pk[1] * g^(-r))
\`\`\`

**EVM (BatchAudit.sol)**:
\`\`\`solidity
// Type-III Asymmetric Pairing, split into G1 × G2
// A = σ_global · g1^{-val}     ∈ G1  (Off-chain aggregation)
// C = [g2]^α · g2^{-r}         ∈ G2  (Off-chain aggregation)
// e(A, g2) · e(-w, C) == 1     (EIP-2537 PAIRING k=2)
\`\`\`

Both are mathematically equivalent: \`e(A, g2) == e(w, C)\` ⟺ \`e(σ·g^{-val}, g) == e(w, g^α·g^{-r})\`

## 4. Experimental Comparison Scheme

### 4.1 Three Major Experiments Comparison

| Experiment | Paper Fig.3 | Python Simulation | EVM Engineering | Output |
|------|-----------|------------|---------|------|
| (e) RANDAO Overhead | 200 executions | \`Blockchain\` random numbers | \`RandAO\` 200 rounds commit-reveal-generate | \`randao_overhead.csv\` |
| (f) Storage Gas | operation a/b/c-d | None (Pure mathematical simulation) | \`FileRootStorage\` + \`BatchAudit\` Real SSTORE | \`storage_gas.csv\` |
| (g) Batch Auditing | T ∈ [10,1000] | \`verify_batch\` timing | \`BatchAudit.verifyBatch\` Real Gas | \`batch_audit.csv\` |

### 4.2 Theoretical Gas Overhead Values (EIP-2537)

| Operation | Precompile | Gas Formula | k=2 Instance |
|------|--------|---------|---------|
| G1MSM (Calculate A) | 0x0c | 12000·k·discount/1000 | ~23808 |
| G2MSM (Calculate C) | 0x0e | 22500·k·discount/1000 | ~44640 |
| PAIRING (Verify) | 0x0f | 32600·k + 37700 | 102900 |
| SSTORE (Initial) | — | 20000 | 20000 |
| SSTORE (Update) | — | 5000 | 5000 |
| SHA-256 (Merkle) | 0x02 | 60 + 12·ceil(len/32) | ~612/execution |

### 4.3 Key Comparison Points

1. **Python simulation** verifies cryptographic correctness (SS512 Type-I), involving no Gas/on-chain overhead.
2. **EVM engineering** verifies on-chain feasibility (BLS12-381 Type-III + EIP-2537), faithfully measuring Gas consumption.
3. Parameters for both are fully aligned (\`config.json\` ↔ \`main.py\`), allowing cross-verification of auditing correctness.
4. EVM engineering additionally measures: RANDAO decentralization overhead, SSTORE storage overhead, and pairing precompile Gas — these represent real on-chain costs that the Python simulation cannot cover.

## 5. Compatibility Notes for Hardhat 2.22.5 and EIP-2537

Hardhat 2.22.5 predates the Prague hard fork (EIP-2537 was introduced in 2.23.0); therefore, the default network lacks native BLS12-381 precompiles at addresses \`0x0b~0x0f\`.

**Solution**: \`BLSPairingShim.sol\` + \`setup_bls_precompiles.js\`

- Inject the shim runtime bytecode into \`0x0f\` via \`hardhat_setCode\`;
- The shim faithfully burns Gas according to the EIP-2537 formula (\`32600·k + 37700\`);
- It defaults to returning a successful pairing (honest audit); tamper testing registers negative cases via \`setForceInvalid\`;
- **On Prague nodes (Hardhat ≥ 2.24), it will automatically fall back to the native precompile, requiring no changes to the contract code.**
